## 4. 开发过程质量评估

### 4.1 Bug 统计与分析

- **总 Bug 数**：5 个。按类型：**逻辑/配置** 3 个（404 因镜像未含新代码、500 表不存在、500 JDBC 未配置）；**语法/类型/构建** 2 个（common 缺 slf4j、logDebug 参数个数错误）；**边界/展示** 0 个；**兼容/环境** 0 个。
- **Bug 发现时机**：联调/用户反馈阶段 3 个（404、500×2）；编码/构建阶段 2 个（slf4j、logDebug）。
- **Bug 修复平均轮次**：约 1～2 轮

### 4.2 典型 Bug 案例剖析

- **Bug1（发信设置 404）**
  - **现象**：打开发信设置或保存均提示失败，接口返回 404。
  - **根本原因**：delivery 镜像使用旧 JAR，未包含 SmtpConfigController。
  - **修复方案**：文档明确「先 mvn package -pl delivery-service -am，再 docker-compose build delivery」；Dockerfile 保持单阶段 COPY 本地 JAR。
  - **预防措施**：CI 或本地在 build 前执行 package；或考虑多阶段构建（本次因网络放弃 Maven 镜像）。

- **Bug2（发信设置 500 - 表不存在）**
  - **现象**：建表前 GET/PUT 返回 500。
  - **根本原因**：租户库未执行 tenant_default-smtp_config.sql。
  - **修复方案**：Controller 捕获异常并识别「doesn't exist」/「smtp_config」，返回明确 errorInfo；文档补充建表顺序与 PowerShell UTF-8 执行示例。
  - **预防措施**：启动/健康检查脚本中校验关键表是否存在；或提供一键初始化 SQL 脚本。

- **Bug3（发信设置 500 - JDBC 连接失败）**
  - **现象**：建表后仍 500，错误为 CannotGetJdbcConnectionException。
  - **根本原因**：docker-compose 中 delivery 未配置 APP_TENANT_BASE_URL 等，仍用 localhost:3306。
  - **修复方案**：为 delivery 增加 APP_TENANT_* 与 depends_on mysql，与 tracking/audit 一致。
  - **预防措施**：所有需连 MySQL 的服务在 compose 中统一检查 base-url 与 depends_on；可写检查清单或脚本校验。

- **Bug4（common 编译失败 - slf4j 不存在）**
  - **现象**：mvn package -pl delivery-service -am 时 common 报 slf4j 不存在。
  - **根本原因**：common 使用 Logger 但 pom 未声明 slf4j-api。
  - **修复方案**：common/pom.xml 增加 slf4j-api 依赖。
  - **预防措施**：凡使用 slf4j 的模块在 pom 中显式声明依赖，不依赖传递。

- **Bug5（logDebug 参数不匹配）**
  - **现象**：delivery 编译报 logDebug 参数类型/个数不匹配。
  - **根本原因**：调试方法 6 参，一处调用只传 4 参。
  - **修复方案**：补全参数；后续移除全部 logDebug。
  - **预防措施**：调试辅助方法用重载或 Map 传参；或尽早移除临时调试代码。

## 5. 项目架构深度评价

### 5.1 扩展性评估

- **改动点统计**：实现「用户 SMTP 配置 + 按创建人发信」共涉及约 18 个文件（含 6 个新增），跨 delivery、campaign、frontend、common、docs、docker-compose。
- **新增代码侵入性**：低。PrepareSendService 仅增加对 campaign.createdBy 的读取与 payload.setSmtpConfigUserId；SendTaskConsumer 增加「若有 smtpConfigUserId 则查 SmtpConfig 并动态发信」分支，原有默认通道逻辑保留。Controller、Service、Mapper 为新增类，对既有接口无侵入。
- **接口友好度**：前端仅需调用 getSmtpConfig()/saveSmtpConfig(data)，表单与错误展示约 80 行；后端 GET/PUT 语义清晰，与现有 /delivery/status 风格一致。

### 5.2 模块合理性评估

- **职责清晰示例**：SmtpConfigService 负责按 user_id 的 CRUD、加密/脱敏；UserSmtpMailSenderFactory 仅负责由 SmtpConfig+明文密码构建 JavaMailSender；SendTaskConsumer 只做「选 sender → 发信 → 回写状态」。
- **职责混淆**：SmtpConfigController 内 mapToBizException 既做「表不存在」友好提示又做通用 500 封装，略混合「业务提示」与「异常转换」，可考虑抽成小型 ExceptionMapper 或工具类。
- **依赖关系**：delivery 依赖 common（TenantContext、Result、BizException）；campaign 仅实体加字段；delivery 通过 DownstreamClient 调 campaign 读 created_by，为单向依赖。未发现循环依赖。

### 5.3 代码质量具体问题

- **重复代码**：SmtpConfigService.getByUserId 与 getEntityByUserId 均为按 user_id selectOne，仅返回 DTO 与 Entity 不同，可保留当前写法或抽成 private 方法返回 Entity 再 toDto。
- **命名**：mapToBizException 名称偏泛，可命名为 toBizExceptionForSmtpConfig 或类似，便于检索。
- **过长函数/类**：SendTaskConsumer.doHandleSendTask 加入「用户 SMTP」分支后约 50 行，仍可读；若再扩展通道类型可考虑拆成「选择 EmailSender」「执行发送」「回写状态」三个小方法。
- **异常处理**：Controller 已捕获 Exception 并转 BizException；Service 层未对 Mapper 异常做业务化分类，仅靠 Controller 根据 message 判断「表不存在」，其他 DB 错误仍为「服务器内部错误」。

### 5.4 架构改进具体建议

- **改进点1**：delivery 在 Docker 下未使用 SPRING_PROFILES_ACTIVE: docker + application-docker.yml，而是直接环境变量覆盖。**建议**：为 delivery 增加 application-docker.yml（app.tenant.base-url 等），compose 中设 SPRING_PROFILES_ACTIVE=docker。**预期收益**：与本地 application.yml 隔离，减少遗漏。
- **改进点2**：发信设置相关 DDL 依赖人工按文档执行。**建议**：在 STARTUP 或运维脚本中提供「租户库完整初始化」命令（按顺序执行所有 tenant_default-*.sql），或健康检查中检测 smtp_config 表存在再标记 ready。**预期收益**：降低漏执行建表导致 500 的概率。
- **改进点3**：「表不存在」的识别依赖异常 message 字符串，对数据库或驱动语言敏感。**建议**：捕获 BadSqlGrammarException 或 SQLException，根据 SQLState 或错误码判断「表/列不存在」，再返回统一 errorInfo。**预期收益**：减少对文案的依赖，更稳定。

## 7. 会话总结

- **本次开发最成功的决策**：① 通过前端 catch 上报 responseData 与后端 500 时附带异常信息到 errorInfo，快速区分 404 与 500（表不存在 vs JDBC 连接失败）；② 用 BUGFIX-LOG 固化 5 条问题的现象、原因、修复与涉及文件，便于后续同类环境复现与排查。
- **本次开发最大的教训**：delivery 在 docker-compose 中未配置 MySQL，与 contact/tracking/audit 不一致，导致「建表后仍 500」。若在交付 Docker 部署时有一份「需连 MySQL 的服务清单 + 必须环境变量与 depends_on」的检查项，可避免此类遗漏。
- **下次类似任务可复用的经验**：① 新接口在 Docker 部署时同步检查该服务的 compose 配置（base-url、depends_on、网络）；② 新增表在文档中明确建表顺序并在常见问题中写「未建表时的报错与解决」；③ 调试用日志/上报在问题关闭后统一清理，保留「对用户可见的友好错误信息」逻辑；④ 编译错误（如缺依赖、方法签名不一致）优先在本地 mvn package 复现并修，再进入镜像构建与部署。
