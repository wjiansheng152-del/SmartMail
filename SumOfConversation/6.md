### 4.2 典型 Bug 案例剖析

- **Bug1（带 Token 创建活动后 created_by 仍为 NULL，发信走 MailHog）**
  - **现象**：POST `/api/campaign/campaign` 已带 `Authorization: Bearer <token>`，但库表 `campaign.created_by` 为 NULL，delivery 按 payload.smtpConfigUserId 取用户 SMTP 时为空，回退到默认通道（MailHog）。
  - **根本原因**：假设为以下之一或组合：网关从 JWT 解析的 userId 为 null 未写入头；网关未将 X-User-Id 转发至 campaign 服务；campaign 收到的 X-User-Id 为空。会话内通过加调试日志定位，未实施代码级修复。
  - **修复方案**：在网关与 campaign 增加 NDJSON 调试日志，复现后根据 userId、xUserIdHeader、createdByBeforeSave 判定断点；根因修复留待后续会话（如网关头透传、JWT 解析类型兼容等）。
  - **预防措施**：网关转发下游时保证自定义头（X-User-Id）透传；JWT 解析时对 userId 做类型兼容（如 number 转 String）；集成测试覆盖「带 Token 创建活动 → 查库 created_by 非空」。

### 4.3 需求变更影响分析

- **需求变更次数**：2 次（见 2.1）。
- **因变更导致的代码重写量**：约 0 行（变更均通过补充文档与加调试代码满足，未推翻已有实现）。

## 5. 项目架构深度评价

### 5.1 扩展性评估

- **改动点统计**：共涉及 4 个文件、2 个模块（gateway、campaign）的修改与 2 个文档的新增。
- **新增代码侵入性**：低。仅在网关 filter 中增加对单一路径的写文件分支、在 campaign 的 create 方法中增加一段写文件；未改路由、未改 DTO/实体；调试代码为临时性质，需在问题解决后移除。
- **接口友好度**：文档中调用方按「登录 → 创建活动（带 Token）→ 创建计划（runAt UTC）」即可操作，步骤与现有 API 一致，无新增接口。

### 5.2 模块合理性评估

- **职责清晰示例**：网关负责鉴权与注入 X-User-Id；campaign 负责按头设置 createdBy 并落库；delivery 按 payload.smtpConfigUserId（来自 campaign.created_by）选 SMTP。
- **职责混淆**：未发现。发信选通道逻辑集中在 SendTaskConsumer，与 PrepareSendService 的 payload 构造一致。
- **依赖关系**：网关 → campaign 为单向；campaign 不依赖 delivery。未发现循环依赖。

### 5.3 代码质量具体问题

- **重复代码**：多处服务（PrepareSendService、ScheduleTriggerService、CampaignTriggerConsumer、DownstreamClient 等）存在类似的「#region agent log + Files.write(Path.of("/tmp/debug-7f1483.log"), ...)」片段，可抽象为统一 DebugLogUtil.write(sessionId, hypothesisId, message, data) 并在问题关闭后统一移除。
- **命名**：无本会话新增命名问题；临时日志文件 debug-7f1483.log 含 sessionId，便于多会话区分。
- **过长函数/类**：JwtAuthFilter.filter 在加入调试分支后略长，但仍在可读范围；CampaignController.create 增加约 10 行后仍为单职责。
- **异常处理缺失**：写调试日志处均为 try/catch 吞异常，避免影响主流程，符合临时调试需求；正式环境应移除或改为可配置的 debug 开关。

### 5.4 架构改进具体建议

- **改进点 1**：网关转发下游请求时，显式配置需要透传的自定义头（如 X-User-Id、X-Tenant-Id），避免依赖默认行为导致头丢失。**方案**：在 Gateway 的 Route 或 Filter 中声明头转发白名单，或使用 AddRequestHeader 等确保上述头一定带出。**预期收益**：可复现地保证 created_by 来源一致，减少环境差异导致的「有时有有时无」。
- **改进点 2**：JWT claim userId 的存储与解析类型统一（IAM 存 String，网关用 String.class 取）；若存在历史或第三方 token 为 number，应在网关侧做兼容（get 后判类型并转 String）。**方案**：网关解析后 `Object uid = claims.get("userId"); String userId = uid != null ? String.valueOf(uid) : null;`。**预期收益**：避免因 claim 类型不一致导致 userId 为 null 进而 X-User-Id 为空。
- **改进点 3**：将「联系人加入分组」从纯 SQL 操作升级为 REST 接口（如 POST `/api/contact/group/{groupId}/members`），与「创建活动必须带 Token」一起写入验证文档。**方案**：在 contact-service 增加上述接口，内部执行 INSERT contact_group_member。**预期收益**：端到端验证可完全通过 API + 文档完成，便于自动化与新人上手。

## 7. 会话总结

- **本次开发最成功的决策**：用数据库查询（smtp_config 与 campaign 表）结合代码引用（SendTaskConsumer、PrepareSendService）直接给出「created_by 为 NULL 导致走 MailHog」的因果链，避免盲目改代码；将「创建活动 + 创建计划」的 POST、Headers、Body 与 PowerShell/SQL 收拢进 SMTP-VERIFY-STEPS-DETAIL.md，便于复现与后续会话接续。
- **本次开发最大的教训**：在用户已明确「带了 token」的情况下，未在当轮通过日志或抓包确认网关是否真的注入并转发了 X-User-Id，而是先加调试日志再等复现，导致修复未在本会话闭环；若在加码前先请用户提供一次请求的响应头或网关日志，可能更快锁定是 JWT 解析还是头转发问题。
- **下次类似任务可复用的经验**：① 先查数据（库表、配置）再查代码，用「数据 + 代码路径」给出可验证的根因假设；② 文档中区分「对外 ID（local_id）」与「用户主键（createdBy）」，并注明 SQL 中表字段对应关系，减少误用；③ 临时状态文档（进展/问题/待办）在暂停或交接时必写，并标明「接续入口」；④ 调试用写文件尽量统一路径与格式（如 NDJSON + sessionId），便于多服务日志合并分析。
