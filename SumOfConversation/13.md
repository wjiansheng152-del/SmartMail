## 5. 项目架构深度评价

### 5.1 扩展性评估

- **改动点统计**：共涉及 3 个前端文件、1 个新增工具模块、4 个文档；未改动后端服务、网关或数据库表结构。
- **新增代码侵入性**：低。仅在 CampaignListView 的 handleSendNow 与 ScheduleListView 的 loadSchedules/handleSubmit 中「包一层」时区转换，未在 API 层或全局拦截器注入逻辑。
- **接口友好度**：对前端开发者透明；创建计划/立即发送调用方式不变，列表字段名不变，仅 runAt 含义与展示值经转换。其它页面可复用 convertUtcStringToLocalDisplay。

### 5.2 模块合理性评估

- **职责清晰示例**：datetime.ts 仅负责无时区字符串↔本地/UTC 换算与格式化；两视图将时区转换委托给 utils，职责清晰。
- **职责混淆**：未发现。
- **依赖关系**：datetime 被两视图引用，无循环依赖；无新外部库。

### 5.3 代码质量具体问题

- **重复代码**：formatUtcRunAtFromLocalDate 与 convertUtcStringToLocalDisplay 中均有「年月日时分秒 padStart 拼 yyyy-MM-dd HH:mm:ss」片段，可抽成内部 formatYmdHms 减少重复，当前可读性尚可，属可选项。
- **命名**：当前命名已表达含义；若后续支持多时区可考虑 utcStringToLocalDisplayInTz 等。
- **过长函数/类**：convertLocalRunAtStringToUtc 约 30 行，可接受；若支持更多格式可拆成「解析→Date」与「Date→UTC 字符串」。
- **异常处理缺失**：格式异常时回退为返回原字符串，未打日志或上报；若需排查用户输入可增加开发环境日志或可选 callback。

### 5.4 架构改进具体建议

- **改进点 1**：verify-smtp-send.ps1 仍用本地时间生成 runAt，Docker 下 scheduler 为 UTC 时会有 8 小时偏差。建议脚本中将「当前本地时间 +2 分钟」转为 UTC 再拼 JSON，或文档注明脚本适用于与 scheduler 同时区、Docker 下建议用手动 POST 或前端创建计划。预期收益：脚本在 Docker 环境下到点触发一致。
- **改进点 2**：后端 schedule 列表返回的 runAt/createTime/updateTime 语义未在接口契约中写明。建议在 API 文档或响应中明确「时间为 UTC，格式 yyyy-MM-dd HH:mm:ss」。预期收益：第三方或脚本直接消费接口时避免误解。
- **改进点 3**：datetime 工具未做单元测试。建议为 datetime.ts 增加单测，覆盖「北京时间 10:00→UTC 02:00」等固定用例。预期收益：防止日后修改时回归。

## 7. 会话总结

- **本次开发最成功的决策**：① 不改后端与表结构，仅在前端统一「提交前转 UTC、展示前转本地」，解决 Docker 下立即发送与创建计划的 8 小时偏差。② 将「前端界面 vs 直接调 API」的时区约定写入 4 份文档，避免 Postman/脚本填错 runAt。
- **本次开发最大的教训**：首次启动命令使用 `cd /d ... && .\mvnw.cmd` 在 PowerShell 下报错；Windows 下应使用 `Set-Location; command` 或分两条命令执行。
- **下次类似任务可复用的经验**：① 涉及时间的接口尽早约定「存储与比较用 UTC、展示用本地」，并在 API 文档与前端工具双处体现。② 「定时不触发」时优先排查调度器时区与请求中时间字符串语义是否一致。③ 时间工具函数集中放 utils、在视图层显式调用，便于测试与复用。④ 文档与代码同步更新，并区分「给界面用户」与「给接口调用方」的说明。
