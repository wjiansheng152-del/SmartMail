- **Bug 发现时机**：联调/用户反馈阶段 5 个。
- **Bug 修复平均轮次**：约 1–2 轮。

### 4.2 典型 Bug 案例剖析

- **Bug1（加入分组后分组内看不到客户）**  
  - **现象**：加入分组返回 200，但「当前成员」与按分组筛选客户为空。  
  - **根本原因**：客户分页接口将 groupId 当作内部主键查 contact_group_member，而写入时使用的是由 local_id 解析出的内部 group id，二者不一致。  
  - **修复方案**：ContactServiceImpl.pageList 中 groupId 先通过 getByLocalIdAndTenant 转为内部 group id 再 selectPageByGroupId。  
  - **预防措施**：对外统一约定「所有分组相关 API 的 groupId 均为 local_id」，并在文档与接口注释中写明。

- **Bug2（立即发送/计划触发到错误活动）**  
  - **现象**：对「活动 id=1」操作时实际投递了别的活动。  
  - **根本原因**：对外 id 为 local_id，delivery 调 campaign 时未带用户信息，campaign 将路径 id 当主键查。  
  - **修复方案**：schedule_job 增加 created_by；CampaignTriggerPayload 与 PrepareSendService 传递 createdBy；DownstreamClient.getCampaign 带 X-User-Id；campaign GET 在存在 X-User-Id 时按 getByLocalIdAndCreatedBy 查询。  
  - **预防措施**：跨服务「按业务 id 查实体」时，若 id 为 local_id，必须同时传递租户/用户维度信息并在被调方按同一维度查询。

- **Bug3（登录后客户/模板/分组/活动 500）**  
  - **现象**：Unknown column 'local_id'。  
  - **根本原因**：代码已切到 local_id，租户库未执行迁移脚本。  
  - **修复方案**：执行 docs/sql 下各 local_id 迁移脚本并重启服务。  
  - **预防措施**：在启动与部署文档中明确「首次或重建库后必须执行迁移脚本」，并在 BUGFIX-LOG 中记录。

## 5. 项目架构深度评价

### 5.1 扩展性评估

- **改动点统计**：约 92 个文件，跨 contact、template、campaign、scheduler、delivery、gateway、frontend、docs 等多模块。
- **新增代码侵入性**：中。在原有「按主键 id」路径上增加「按 local_id（+ 租户/用户）」分支，侵入点包括各 Controller 路径参数解析、Service 的 getByLocalId/nextLocalId、Mapper 按 local_id 查询、delivery 与 scheduler 的 payload 与 HTTP 头；未在核心领域模型内引入大量 if/else，通过「对外统一用 local_id、内部按需解析」收敛。
- **接口友好度**：前端与调用方仅需使用接口返回的 id（local_id）和创建计划时传 createdBy；SMTP 文档明确「参数需要的 id 类型」后，联调成本降低。

### 5.2 模块合理性评估

- **职责清晰示例**：contact-service 内 ContactGroupMemberService 专注分组成员关系，ContactServiceImpl 负责客户分页与 groupId 解析；scheduler 负责计划 CRUD 与触发，delivery 负责拉取活动与发信，createdBy 通过 payload 与 HTTP 头传递，边界明确。
- **职责混淆**：未发现代码层面职责混淆；文档层面曾出现「把联系人加入分组」既写 SQL 又应写接口的混用，已通过「接口优先、SQL 备选」区分。
- **依赖关系**：delivery 依赖 campaign/contact 的 HTTP 接口，scheduler 投递 MQ 后 delivery 消费，单向依赖；无循环依赖。

### 5.3 代码质量具体问题

- **重复代码**：各服务中「按 local_id 查实体」的模式类似，可考虑在 common 或各模块内抽象为统一工具；当前各服务自行实现，尚可接受。
- **命名**：ScheduleJobListItem 与 CampaignListItem、ContactListItem 一致；createdBy 在 payload 与请求体中统一，PrepareSendService 中曾用 campaignCreatedBy 区分参数与响应字段。
- **过长函数/类**：ContactServiceImpl、CampaignServiceImpl 在增加 local_id 与解析逻辑后行数增多，未明显超长；后续若再增加「按某维度 id 解析」逻辑，建议抽成私有方法或独立 Helper。
- **异常处理缺失**：getByLocalIdAndTenant 或 getByLocalIdAndCreatedBy 查不到时，部分路径需统一为 404 与明确 errorInfo。

### 5.4 架构改进具体建议

- **改进点 1**：**问题**——多处「按 local_id 解析为内部 id」的逻辑分散在各 Service。**建议**——在 common 或各模块内提供「resolveLocalIdToInternalId(entityType, localId, tenantIdOrUserId)」的约定或工具。**预期收益**：新增类似实体时复用同一套解析与错误处理。
- **改进点 2**：**问题**——SMTP 验证文档与 API 文档中关于「contact_group_member 用内部主键」的说明分散。**建议**——在 API-AND-CLASS-REFERENCE 或单独「数据模型与 ID 约定」中集中说明表内主键与对外 local_id 的对应关系，SMTP 文档仅引用。**预期收益**：减少文档不同步与用户困惑。
- **改进点 3**：**问题**——迁移脚本与基础建表脚本分离，新环境需按顺序执行两套脚本，易漏。**建议**——在保证幂等的前提下，提供「一键初始化」脚本或 Docker 初始化入口，或在文档中明确「若从零建库，必须执行迁移脚本」的清单。**预期收益**：降低部署与排错成本。

## 7. 会话总结

- **本次开发最成功的决策**：① 在文档中统一增加「ID 映射与参数说明」表格，明确 local_id、用户 ID、内部主键在不同场景下的用法。② 将「将联系人加入分组」改为推荐 REST 接口、SQL 备选，使文档与实现一致。③ 在 BUGFIX-LOG 与 STARTUP 中明确迁移脚本列表与执行顺序，便于新环境部署与 500 排查。
- **本次开发最大的教训**：① 文档修改后未做「是否覆盖所有约定表述」的自检，导致用户反馈「没有看到修改」时才发现「接口优先」段落未写入。② 「当前没有 REST 接口」类表述易与实现脱节，应在需求或实现变更时同步检索文档并更新。
- **下次类似任务可复用的经验**：① 涉及「对外 id」与「内部主键」分离时，在接口与验证类文档中尽早写清「各参数/表字段对应的 id 类型」并配表格。② 涉及「可选实现方式」（如接口 vs SQL）时，明确主推方式并写清备选适用场景。③ 多文件文档更新时用「变更清单」逐项确认（如 ID 映射表、createdBy、接口优先、SQL 备选），减少漏改。
