# 会话总结模板与统一标准

本文档定义 SmartMail 项目「会话开发总结」的**统一模板、量化标准与判断口径**，便于各会话 AI 按同一规范产出或修订总结，保证跨会话可比、可审计。

---

## 一、文档用途与使用方式

- **模板**：下方第二～八节为固定章节结构，各会话总结必须完整包含且顺序一致。
- **量化标准**：第三～八节中涉及数字、分类、表格的字段，按「二、量化与判断标准」执行，避免各会话自创口径。
- **向会话 AI 提修订需求时**：可直接复制「九、给对应会话 AI 的提示词」中的整段，粘贴到该会话中，要求其按本模板与标准修改已有总结。

---

## 二、量化与判断标准（统一口径）

### 2.1 会话基础信息

| 字段 | 填写规则 | 示例 |
|------|----------|------|
| **会话总轮数** | 只填数字或「约 X 轮」；一轮 = 用户一次完整输入 + AI 一次完整回复（含多轮工具调用计为 1 轮）。无法统计时写「未统计」，不得留空或写「需从会话日志统计」。 | `约 25 轮` / `8 轮` / `未统计` |
| **会话总字符数** | 格式：`用户输入约 X 字符 / AI 输出约 Y 字符`。无法统计时写「未统计」，不得用占位符或「需从会话日志统计」。估算时注明「按可见对话估算」。 | `用户输入约 4000 字符 / AI 输出约 35000 字符（按可见对话估算）` |
| **会话持续时间** | 填写「约 X 分钟」或「约 X 小时」或「未记录」。不写「需从会话日志统计」。 | `约 2–3 小时` / `约 45 分钟` / `未记录` |
| **本次对话核心目标** | 用编号列出 1～5 条，每条一句话；仅写「本会话内」达成的目标，不写项目长期目标。 | `① 解决 Docker 报错「container is not running」；② 按模板产出会话 N 的全面分析总结。` |
| **最终达成结果** | 用 1～3 句话概括：哪些目标完成、哪些部分完成或未完成；若有未完成项，必须明确写出。 | `① 已解释报错含义与处理步骤；② 已按模板撰写并落盘总结至 SumOfConversation/N.md。` |

### 2.2 沟通效率

| 字段 | 填写规则 |
|------|----------|
| **需求明确轮次** | 填达到「目标与范围清晰」所需的**轮数**（数字或「约 X 轮」），不是字数。 |
| **需求变更次数** | 填**次数**（0、1、2…）；每次变更需在下列出：变更内容（从 A 改为 B）、触发原因。 |
| **无效对话轮次** | 填**轮数**：完全无推进、仅重复澄清、或仅回复「再试一次」等无实质产出的轮次。 |

### 2.3 代码变更统计

| 字段 | 填写规则 |
|------|----------|
| **新增/修改/删除代码行数** | 必须区分：**核心逻辑**、**配置/样板/注释**、**文档**；行数可为约数，格式如「约 380 行（核心逻辑约 120 行 / 配置与注释约 260 行）」。无代码变更时写「0 行」并说明「本会话未修改项目代码」。 |
| **新增/修改/删除文件数** | 填具体数字；总结文档（如 SumOfConversation/N.md）计为「新增文件」，但不计入「业务/配置代码」文件数说明。 |

### 2.4 Bug 分类与统计（统一类型）

Bug 类型**仅**使用以下四类，并在括号中注明子类（可选）：

1. **逻辑/配置**：业务逻辑错误、配置错误、数据一致性问题、环境/部署配置导致的现象。
2. **语法/类型/构建**：编译错误、类型错误、缺少依赖、导出/导入错误。
3. **边界/展示**：边界条件、UI 展示错误、样式或布局问题。
4. **兼容/环境**：运行环境、时区、编码、依赖版本、启动顺序等导致的兼容性或环境问题。

| 字段 | 填写规则 |
|------|----------|
| **总 Bug 数** | 数字；按「逻辑/配置」「语法/类型/构建」「边界/展示」「兼容/环境」分别统计并汇总。无 Bug 时写「0 个」并注明「本会话未引入/未修复代码缺陷」或「问题根因为使用方式/环境，非代码缺陷」。 |
| **Bug 发现时机** | 仅允许：`编码/构建阶段`、`联调/用户反馈阶段`、`不适用`。 |
| **Bug 修复平均轮次** | 数字或「约 X 轮」；未修复写「0 轮」或「不适用」。 |

### 2.5 开发周期阶段名称（统一）

表格「开发周期分解」中**阶段**列仅使用以下五项（按顺序），不得自创：

1. **需求澄清**
2. **方案设计**
3. **编码/配置实现**
4. **测试修复**
5. **文档/收尾**（或「文档与总结」）

### 2.6 需求匹配度

- **完全实现的需求点**：列出条目，与「本次对话核心目标」对应。
- **部分实现的需求点**：明确写出「哪一条目标」部分完成及原因。
- **超出范围实现的点**：列出并注明「合理扩展」或「必要补充」。
- **最初需求与最终产出的差异分析**：用 1～3 句话说明匹配程度与主要差异。

### 2.7 不适用时的写法

当某节对本会话不适用时（例如无代码变更、无 Bug）：

- 在**小节标题**下写一句说明，再写「不适用」或具体结论。  
- 示例：「本会话未修改项目代码，不涉及代码复用统计。」「本会话无编码与联调，Bug 发现/修复轮次不适用。」

禁止为凑篇幅编造数据；禁止使用「___」「待填」「需从会话日志统计」等占位符。

## 三、固定章节结构（模板正文）

以下为各会话总结**必须包含**的章节与子节，顺序不可调换；子节可根据会话内容合并为「不适用」说明，但节号与标题需保留。

### 1. 会话基础信息

- 会话总轮数
- 会话总字符数
- 会话持续时间
- 本次对话核心目标
- 最终达成结果

### 2. 对话内容深度分析

#### 2.1 沟通效率统计

- 需求明确轮次
- 需求变更次数（每次：变更内容、触发原因）
- 无效对话轮次

#### 2.2 信息质量分析

- 关键信息点提取（用户侧）
- AI 关键贡献
- 信息断层实例（无则写「无显著断层」并简述原因）

### 3. 代码变更详细记录

#### 3.1 代码变更统计

- 新增代码行数（含核心逻辑/配置与注释/文档的拆分）
- 修改代码行数
- 删除代码行数
- 新增文件数 / 修改文件数 / 删除文件数

#### 3.2 关键文件变更明细

**表格列名（统一）：**

| 文件路径 | 变更类型 | 行数变化 | 变更目的/作用 |
|----------|----------|----------|----------------|

- **变更类型**：仅填「新增」「修改」「删除」之一。
- **行数变化**：如 `+约 22`、`-4 +1`、`+约 10`。

#### 3.3 代码复用情况

- 直接复用的现有代码/组件
- 借鉴现有模式实现的代码
- 完全新写的代码（占比约 X%）

（无代码变更时：写「本会话未涉及业务代码编写，不适用代码复用统计。」）

### 4. 开发过程质量评估

#### 4.1 Bug 统计与分析

- 总 Bug 数（按 2.4 节类型分类）
- Bug 发现时机
- Bug 修复平均轮次

#### 4.2 典型 Bug 案例剖析

每个 Bug 统一格式：

- **BugN（简短现象描述）**
  - **现象**：……
  - **根本原因**：……
  - **修复方案**：……
  - **预防措施**：……

（无 Bug 时：写「本会话无代码 Bug 案例。」并可补充「问题澄清」类说明。）

#### 4.3 需求变更影响分析

- 需求变更次数
- 因变更导致的代码重写量（行数）
- 变更管理评价（1～2 句话）

### 5. 项目架构深度评价

#### 5.1 扩展性评估

- 改动点统计（涉及文件数、模块数）
- 新增代码侵入性（低/中/高，用实例说明）
- 接口友好度（不涉及接口时可写「不适用」）

#### 5.2 模块合理性评估

- 职责清晰示例 / 职责混淆（无则写「未发现」）
- 依赖关系（无循环依赖需明确写出）

#### 5.3 代码质量具体问题

- 重复代码、命名、过长函数/类、异常处理缺失（每项有则简述，无则写「无」或「不适用」）

#### 5.4 架构改进具体建议

- 改进点 1～3：**问题** → **方案/建议** → **预期收益**（各 1 句话）。无则写「本会话未进行架构级开发，无基于本次会话的架构改进建议。」

### 6. 需求落地效率分析

#### 6.1 开发周期分解

**表格列名（统一）：**

| 阶段 | 耗时(轮数) | 产出 | 卡点/问题 |
|------|------------|------|-----------|

阶段仅使用：需求澄清、方案设计、编码/配置实现、测试修复、文档/收尾。

#### 6.2 代码有效性分析

- 核心必要代码（行数，占比）
- 配置/样板代码（行数，占比）
- 冗余/未使用代码（行数）
- 过度设计代码（行数）

（无代码时四项均为 0 行 / 0%。）

#### 6.3 需求匹配度评估

- 完全实现的需求点
- 部分实现的需求点
- 超出范围实现的点
- 最初需求与最终代码的差异分析

### 7. 会话总结

- **本次开发最成功的决策**：（1～3 条，每条一句话或短句。）
- **本次开发最大的教训**：（1～2 条；无则写「无显著教训」并简述原因。）
- **下次类似任务可复用的经验**：（1～4 条，具体可操作。）

---

## 四、标题与文件命名规范

- **文档标题**：统一为 `# 会话N 开发总结`（N 为会话编号，与文件名一致；「会话」与数字之间可留一空格：`会话 N 开发总结`）。不使用「开发会话总结」「开发总结报告」等变体。
- **文件名**：`N.md`，存放于 `SumOfConversation/` 目录下（即 `SumOfConversation/N.md`）。不要写成 `docs/SumOfConversation/N.md`。

---

## 五、给对应会话 AI 的提示词（复制使用）

将下面整段复制到**需要修订总结的那次会话**中，发给该会话的 AI：

---

请你以「项目分析师」身份，根据本会话的完整对话与最终代码/文档结果，对**本会话的总结文档**做一次修订，使其符合项目统一标准。

**请严格按以下要求执行：**

1. **使用统一模板与结构**  
   总结必须包含且仅包含以下 7 个大节，顺序不可变：  
   **1. 会话基础信息** → **2. 对话内容深度分析** → **3. 代码变更详细记录** → **4. 开发过程质量评估** → **5. 项目架构深度评价** → **6. 需求落地效率分析** → **7. 会话总结**。  
   各节子节（如 2.1 沟通效率统计、3.2 关键文件变更明细、4.2 典型 Bug 案例剖析、6.1 开发周期分解等）需完整保留，与项目内 `SumOfConversation/会话总结模板与统一标准.md` 中「三、固定章节结构」一致。

2. **遵守量化与判断标准**  
   - **会话基础信息**：会话总轮数、总字符数、持续时间若无法精确统计，写「约 X 轮」「约 X 字符」「未记录」或「未统计」，不得使用「___」「需从会话日志统计」等占位符。  
   - **Bug 类型**：仅使用四类——**逻辑/配置**、**语法/类型/构建**、**边界/展示**、**兼容/环境**；在 4.1 中按类型统计并汇总总 Bug 数。  
   - **开发周期阶段**：6.1 表格中「阶段」列仅使用：需求澄清、方案设计、编码/配置实现、测试修复、文档/收尾。  
   - **关键文件变更明细表**：列名为「文件路径」「变更类型」「行数变化」「变更目的/作用」；变更类型仅填「新增」「修改」「删除」之一。

3. **不适用时的写法**  
   若本会话无代码变更、无 Bug、未涉及架构改动等，在对应小节下用一句话说明原因后写「不适用」或给出简要结论，不要编造数据或为凑篇幅填充无关内容。

4. **标题与输出**  
   - 文档标题统一为：`# 会话N 开发总结`（N 为本会话编号）。  
   - 修订后的完整总结请输出到仓库内 `SumOfConversation/N.md`（N 为本会话编号），不要使用 `docs/SumOfConversation/` 路径。

请基于当前会话的真实内容与结果进行修订，保留所有有依据的量化数据和具体案例，仅统一格式、口径和缺失项写法，使本会话总结与其它会话总结可比、可审计。

---

（以上为可直接发送给对应会话 AI 的完整提示词。）

|--------|--------|
| 标题为「会话N 开发总结」且文件名为 `SumOfConversation/N.md` | |
| 7 大节齐全且顺序与模板一致 | |
| 会话总轮数/字符数/持续时间无占位符（___、需从会话日志统计等） | |
| Bug 仅使用四类：逻辑/配置、语法/类型/构建、边界/展示、兼容/环境 | |
| 开发周期阶段仅使用规定的 5 个阶段名 | |
| 关键文件变更表列为：文件路径、变更类型、行数变化、变更目的/作用 | |
| 开发周期表列为：阶段、耗时(轮数)、产出、卡点/问题 | |
| 无代码/无 Bug 时对应小节有「不适用」或简要说明，无编造数据 | |
