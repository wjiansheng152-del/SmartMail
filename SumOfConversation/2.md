## 4. 开发过程质量评估

### 4.1 Bug 统计与分析

- **总 Bug 数**：4 个。按类型：**逻辑/配置** 1 个、**语法/类型/构建** 2 个、**边界/展示** 1 个。
- **Bug 发现时机**：编码/构建阶段 3 个、联调/用户反馈阶段 1 个。
- **Bug 修复平均轮次**：约 1 轮。

### 4.2 典型 Bug 案例剖析

- **Bug1（模板/分组下拉无数据）**
  - **现象**：CampaignListView 中模板与分组下拉为空，新建活动无法选择。
  - **根本原因**：loadOptions 误写为两次 getCampaignList，未调用 getTemplateList / getGroupList。
  - **修复方案**：改为调用 templateApi.getTemplateList()、contactApi.getGroupList()。
  - **预防措施**：按模块调用对应 api 文件，避免复制粘贴后未替换接口名。

- **Bug2（request 模块无 default 导出导致构建失败）**
  - **现象**：前端构建报错「Module request has no default export」。
  - **根本原因**：request.ts 仅 export const request，多处使用 import request from '@/utils/request'。
  - **修复方案**：增加 export default requestInstance 及 export { requestInstance as request }。
  - **预防措施**：对外统一使用 default 导出，或全项目统一改为命名导入。

- **Bug3（Vuex 无声明文件导致类型报错）**
  - **现象**：TypeScript 报「Could not find a declaration file for module 'vuex'」。
  - **根本原因**：Vuex 4 的 package.json exports 未暴露类型，ts 解析到 .mjs。
  - **修复方案**：在 env.d.ts 中 declare module 'vuex' 并补充 Store/useStore/createStore 类型及 install。
  - **预防措施**：对类型未暴露或 exports 不友好的依赖，在项目内补写声明。

- **Bug4（主内容区不填满浏览器）**
  - **现象**：主内容区右侧大面积空白，仅左侧小卡片可见。
  - **根本原因**：main.css 中 #app 的 max-width:1280px、margin:0 auto 及媒体查询中 grid 与 body 居中。
  - **修复方案**：删除限宽与媒体查询，改为 #app 的 width:100%、min-height:100vh。
  - **预防措施**：后台类项目在接入主布局后立即检查全局 CSS 对 #app/body 的约束。

### 4.3 需求变更影响分析

- **需求变更次数**：1 次（策略与目录名确认）。
- **因变更导致的代码重写量**：0 行（变更在实现前确认，未造成返工）。
- **变更管理评价**：变更为必要且及时，在首轮计划中即明确「占位」与目录名，避免实现到一半再改。

## 5. 项目架构深度评价

### 5.1 扩展性评估

- **改动点统计**：本会话涉及新增 frontend 整仓（约 32 个有效文件）、修改约 12 个文件（含 2 个后端/文档/编排）；若仅算业务模块则约 8 个视图、8 个 api 模块、2 个 store 模块。
- **新增代码侵入性**：低。前端为零侵入（独立 frontend 目录）；后端仅在 CampaignController 增加一个 PUT 方法；docker-compose 仅追加 frontend 服务。
- **接口友好度**：调用方使用新功能为「路由 + 页面内按钮/表单」，API 层按模块分文件，调用处 1～2 行即可完成请求。

### 5.2 模块合理性评估

- **职责清晰示例**：src/api/*.ts 按领域划分与后端服务一一对应；src/store/modules/user.ts 仅负责登录态与 Token；request 通过 AuthProvider 解耦 store。
- **职责混淆**：CampaignListView 内同时拉取 templateList、groupList 与 campaign list，若多处需要「模板+分组」下拉可抽成 composable 或公共 store；DashboardView 内直接 request.get 多个 list，可抽成 dashboardApi。
- **依赖关系**：无循环依赖；request 依赖由 main.ts 注入的 AuthProvider，store 依赖 request，依赖方向单一。

### 5.3 代码质量具体问题

- **重复代码**：各列表页的 loadList + loading + 表格结构模式重复，可抽成 useListTable 或通用 ListPage 布局。
- **命名**：MainLayout 中 titleMap 的 key 与 router path 一致，命名清晰；无显著命名问题。
- **过长函数/类**：ContactListView 单文件约 245 行，含三个 Tab 与两个弹窗，可拆成三个子组件。
- **异常处理**：列表接口失败时部分页面仅静默置空，未统一 ElMessage 提示；网络错误未做二次提示。

### 5.4 架构改进具体建议

- **改进点 1**：多页重复「loading + 列表 + 分页/刷新」逻辑 → 抽 useListLoader 或通用 ListContainer，统一 loading/error 与空状态 → 减少重复、统一展示。
- **改进点 2**：非 401 的业务错误用户感知不足 → 在 request 响应拦截器中对 4xx/5xx 统一读取 errorCode/errorInfo 并 ElMessage.error → 接口异常时用户能明确看到提示。
- **改进点 3**：审计列表需兼容 Result<List> 与 Result<IPage> 两种形态 → 与后端约定列表类接口统一为 IPage 或 List，前端只处理一种 → 类型简单、少写分支。

## 7. 会话总结

- **本次开发最成功的决策**：① 先产出计划并明确后端缺口与「占位/先做前端」两种路径，由用户选择后再实现，避免反复改需求；② 「立即发送」用「创建 runAt=当前+1 分钟」的调度计划实现，复用现有调度与投递链路，无需新接口；③ 通过 AuthProvider 注入将 request 与 store 解耦，避免循环依赖。
- **本次开发最大的教训**：全局样式（main.css）继承自脚手架，未在接入 MainLayout 时同步清理 #app 的 max-width 与媒体查询，导致「主区域不填满」的体验问题；应在接入主布局后立即检查根节点与 body 的宽高和布局方式。
- **下次类似任务可复用的经验**：① 前后端分离项目先列「后端已有/缺失接口」再定前端范围与占位策略；② 新前端工程在接入布局后马上检查全局 CSS 对 #app/body 的约束；③ 对 Vuex 等类型未暴露完整的依赖，在 env.d.ts 中集中声明；④ 列表/表单类页面可先做 1～2 个标杆页，再按同一模式复制并替换 API/字段，减少重复与遗漏。
