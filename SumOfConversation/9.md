## 5. 项目架构深度评价

### 5.1 扩展性评估

- **改动点统计**：实现本次功能涉及约 20 个文件、4 个模块（contact-service、campaign-service、frontend、docs）。
- **新增代码侵入性**：低。ContactGroupController 仅新增 4 个方法，未改原有 create/get/list/delete；ContactController 新增 1 个 update 方法；CampaignServiceImpl 仅新增 removeCampaign 及一处调用。前端在 ContactListView 内增加计算属性与表单项，未动路由或全局状态。
- **接口友好度**：调用方使用新功能成本低。例如前端「加入分组」：单条调用 addMemberToGroup(groupId, contactId)，批量调用 addMemberToGroupBatch(groupId, contactIds)，与后端 path/body 一一对应，符合直觉。

### 5.2 模块合理性评估

- **职责划分清晰**：ContactGroupMemberService 只负责分组成员的增删与存在性校验；ContactService 负责客户 CRUD，update 与 getByLocalIdAndTenant 归属同一层，便于复用。
- **职责混淆**：未发现。ContactGroupController 同时承担「分组 CRUD」与「分组成员」接口，同属分组域，可接受。
- **依赖关系**：ContactGroupMemberServiceImpl 依赖 ContactGroupService、ContactService、ContactGroupMemberMapper，单向依赖，无循环；campaign 删除仅依赖 CampaignAbAssignmentMapper 与 CampaignMapper，无新增循环依赖。

### 5.3 代码质量具体问题

- **重复代码**：ContactGroupMemberServiceImpl 中 addToGroup 与 addToGroupBatch 内「解析 group/contact、查 existing、insert」逻辑相似，可抽成 private 方法（如 ensureMember）减少重复。
- **命名**：无突出问题；RemoveGroupMemberBatchRequest 与 AddGroupMemberBatchRequest 命名一致、表意清晰。
- **过长函数/类**：ContactListView.vue 体量较大（约 770 行），客户列表与分组管理/编辑弹窗均在同一文件，后续可考虑拆成三个子组件以利维护。
- **异常处理缺失**：前端对 409（邮箱冲突）已接；后端批量移出时对「部分 contactId 不存在」采用静默过滤，若需「全部不存在则报错」可再明确约定并实现。

### 5.4 架构改进具体建议

- **改进点 1**：ContactGroupMemberServiceImpl 内单条/批量加入逻辑重复。**方案**：抽取 ensureMember(Long internalGroupId, Long internalContactId)，addToGroup 调用一次，addToGroupBatch 循环调用。**预期收益**：减少重复，后续改「唯一约束冲突」处理时只改一处。
- **改进点 2**：ContactListView.vue 单文件过大。**方案**：拆成 ContactListTab、GroupManageTab、GroupEditDialog 等子组件，通过 props/emit 或 provide/inject 传分组列表与刷新回调。**预期收益**：便于单测与多人协作，减少合并冲突。
- **改进点 3**：DELETE 带 body（removeMemberFromGroupBatch）在部分网关或规范中不常见。**建议**：保持当前实现，在 API 文档中明确「DELETE .../member/batch 请求体为 { contactIds }」；若未来统一为仅 path/query，可再增加 query 或改为 POST .../member/batch/remove。

## 7. 会话总结

- **本次开发最成功的决策**：① 分组成员与客户更新统一使用 local_id 作为对外 id，在 Service 层统一解析为内部主键，与现有 contact/group/campaign 的 local_id 策略一致。② 删除活动时在应用层显式先删 campaign_ab_assignment 再删 campaign，符合「禁止外键与级联」规约且行为清晰。
- **本次开发最大的教训**：前端大块模板编辑时误删「新建分组」弹窗的 el-dialog 开始标签，导致编译报错；大块替换时应保留关键结构或先读再改，避免仅保留闭合标签。
- **下次类似任务可复用的经验**：① 新增「对已有资源的关联管理」（如分组成员）时，优先沿用现有「local_id 入参 + Service 内解析」模式。② 删除带关联表的资源时，在应用层先删关联再删主表并写进接口文档。③ 列表页排序/搜索若仅针对当前数据，用前端 sort+filter 即可，避免与分页语义混淆。④ 会话结束前将「会话总结 + 关键文档更新」列为明确交付项，减少收尾轮次补充。
