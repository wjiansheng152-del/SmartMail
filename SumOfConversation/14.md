#### 4.2 典型 Bug 案例剖析

- **Bug1（注册账号 down 后再 up 无法登录）**
  - **现象**：docker-compose down 再 up 后，原注册账号无法登录，再次注册同用户名后可登录且能看到原模板数据。
  - **根本原因**：IAM 在 Docker 中通过环境变量使用 jdbc:h2:mem:iam，数据仅存于容器内存、无卷；业务数据在 MySQL 的 mysql_data 卷上故保留；同一用户名再注册得到相同 tenant_id 故业务数据「看起来仍在」。
  - **修复方案**：IAM 在 Docker 下改为连接 MySQL 的 iam 库（application-docker.yml + docker-compose 环境变量），JPA ddl-auto: update，不新增卷。
  - **预防措施**：部署前明确各服务数据存储位置（内存 vs 持久卷），对需持久化的服务避免内存库或为内存库配卷。

- **Bug2（首次 up 后立刻登录失败）**
  - **现象**：首次执行 docker-compose up -d 后马上调登录接口失败，多次 up 或等待一段时间后恢复正常。
  - **根本原因**：depends_on: mysql healthy 仅保证 MySQL 进程可 ping，不保证立即能接受建库/建表等连接；IAM 启动时 Hikari 默认 30s 连接超时，若此时 MySQL 仍忙则连接失败。
  - **修复方案**：在 application-docker.yml 中设置 spring.datasource.hikari.connection-timeout: 60000。
  - **预防措施**：对强依赖数据库的服务可统一拉长 Hikari 连接超时或增加启动重试/健康检查，并在文档中说明「首次启动后若登录失败可等待数十秒」。

#### 5.4 架构改进具体建议

- **改进点 1**：各服务均依赖 MySQL，若需统一「等待 DB 就绪」策略，可在各服务 application-docker.yml 中统一 Hikari connection-timeout（如 60000），或引入 Spring Boot 的 datasource 初始化重试。**预期收益**：降低首次 up 后需多次重启或等待的几率。

- **改进点 3**：docker cp 取容器内日志依赖容器名（如 smartmail-iam-1），不同环境可能不同；可在文档或脚本中统一写「先 docker ps 查 IAM 容器名再 cp」，或提供按镜像名查找容器再 cp 的示例。**预期收益**：排查与日志收集更可复现。

## 7. 会话总结

- **本次开发最成功的决策**：① 用「H2 内存 vs MySQL 卷」和「同一用户名→同一 tenant_id」清晰解释现象，避免误判为业务逻辑 Bug；② 采用 Debug 模式做假设与打点，用日志确认 user_found + afterAuth 和 DataInitializer 正常，将问题收敛到启动顺序，再针对性加 Hikari 超时；③ 不改动 IAM 核心逻辑，仅通过 profile 配置与 docker-compose 环境变量完成持久化与超时，侵入性小、易回滚。
- **本次开发最大的教训**：① 若用户首轮即说明「有时刚启动登录不可用」，可更早把「启动顺序/连接超时」纳入方案，减少中段调试轮次；② 在文档或复现步骤中应明确「先触发登录再 docker cp 日志」以及容器名获取方式，减少环境差异导致的无效尝试。
- **下次类似任务可复用的经验**：① 涉及「down 后数据丢失」时，先区分「数据存哪里（内存/卷）」和「谁在用谁」，再决定是加卷还是换数据源；② Docker 下服务启动顺序问题，除 depends_on 外可配合拉长连接超时或重试；③ 临时调试代码在验证后应彻底移除并同步更新文档；④ 多份文档（状态、启动、后端总结、Bug 记录）同步更新，便于后续排查与交接。
