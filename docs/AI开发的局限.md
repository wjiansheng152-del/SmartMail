# AI 开发的局限

本文档概括在软件开发中采用 AI 辅助时仍存在的技术边界与局限，便于团队设定合理预期并配合流程缓解。

---

## 1. 说明与分类

尽管 AI 能够显著提升效率，仍然存在明显技术边界。以下说明基于当前主流模型能力：

- **部分历史上高发的问题已随 AI 发展明显缓解**，不必再当作默认高发风险。
- **另有一部分可通过使用者的流程与操作缓解**。

---

## 2. 已明显缓解的问题（当前较少出现）

以下类型在现有主流模型下已大大缓解，但仍建议在关键链路上做常规抽查：

| 类型 | 说明 |
|------|------|
| 逻辑错误、复制粘贴残留 | 模型对上下文与一致性的把握增强，此类问题明显减少。 |
| 环境假设错误 | 当前 AI 会意识到本地/Docker/多环境差异，并主动提示使用者确认。 |
| 数据与迁移 | 对建表、迁移脚本与代码的配套理解更好，遗漏迁移的情况减少。 |
| 敏感信息泄露 | 多数模型会优先采用配置/环境变量而非硬编码，并提示使用者注意安全。 |
| API/库幻觉 | 对真实 API 与文档的依从性提高，虚构接口或参数的情况减少。 |
| 权限与越权 | 对鉴权、权限边界的理解增强，此类设计错误有所缓解。 |

---

## 3. 可通过使用者操作缓解的问题

| 问题 | 缓解方式 |
|------|----------|
| **需求理解偏差** | 动工前先充分沟通、冻结契约；可使用 Cursor 的 **Plan 模式** 等先对齐再开发。 |
| **一次性大改** | **限制单次 AI 改动范围**，**逐步、逐接口验证**，避免一次改一大片再集中联调。 |
| **知识滞后** | **及时更新项目文档与依赖**，或**使用较新版本的模型/工具**，减轻训练数据滞后影响。 |

---

## 4. 仍须重视的局限

**上下文窗口限制**：长任务中可能出现上下文丢失、多模块理解偏差、隔天需重新描述背景。应对：人工拆解任务、模块化设计、清晰的项目结构。

**代码质量与架构**：仍可能出现不合理架构与过度耦合、边界与异常路径考虑不足、与项目既有约定不一致。应对：AI 生成代码须经人工审查与验收，最终正确性由工程师负责。

**对开发者能力要求更高**：开发者需具备架构设计、需求表达、技术判断、问题分析能力，能判断 AI 方案是否合理、修正错误代码、优化设计。

**复杂 UI/复杂问题**：复杂 UI 往往需要多轮调整；非常复杂的需求若未先由人拆解，AI 可能只实现一部分或缺乏系统结构。应对：人工先拆任务、明确验收与非功能需求。

---

## 5. 责任边界

- AI 不承担最终业务正确性责任，最终责任由 **AI 使用者**承担。
- 凡「出问题需要背责」的决策（发布、回滚、数据、安全）必须由岗位负责人主导，AI 只可辅助。
