# AI 开发的成本问题

本文档说明在使用 AI 进行开发时需要考虑的成本构成及控制建议，便于团队在「效率提升」与「成本可控」之间做出合理权衡。

---

## 1. 成本构成

### 1.1 直接金钱成本

- **订阅费用**：IDE 或编程助手的会员费（如 Cursor Pro、GitHub Copilot 等）。
- **API 调用费用**：若通过 API 使用大模型（OpenAI、Claude、国产大模型等），按 token 或按次计费。
- **企业版/团队席位**：团队统一采购时的额外费用。

### 1.2 用量（Token）成本

- **输入（Input）**：每次请求会带入代码、文档、对话历史、需求卡等，上下文越长，单次请求成本越高。
- **输出（Output）**：生成的代码、解释、多轮回复均计入输出 token，单价通常高于输入。
- **多轮与重试**：理解错误导致的反复重试、大范围「重来」会显著放大总用量。

### 1.3 间接/机会成本

- 纠正 AI 错误、排查 AI 引入问题所花费的时间。
- 为让 AI 更好用而维护文档、写清需求、整理 rules 的时间。
- 过度依赖且不做人工验收时，可能弱化对契约与边界的敏感度，带来后续返工或线上问题。

---

## 2. 成本控制建议

| 方面 | 建议 |
|------|------|
| **控制上下文** | 不无差别 @ 整个仓库；只提供与当前任务相关的文件或目录，必要时用精简的接口/契约摘要代替大段代码。 |
| **任务拆小、提示写清** | 单任务、单次改动，减少「理解错了再重来」的轮次，从而降低总 token 与时间成本。 |
| **用好 rules/skills** | 把规范、契约、常用约束写进项目 rules 或 skills，避免每次对话都大段重复说明，既省 token 又减少偏差。 |
| **按任务选模型** | 简单补全、格式整理等用更快/更便宜的模型；复杂设计、跨模块改动再用能力更强的模型。 |
| **设定预算或用量意识** | 若走 API，可设月度/项目预算或用量告警；若用订阅制，对「重任务」会拉高单日用量有基本认知。 |

---

## 3. 与质量、流程的关系

在满足质量与契约的前提下，**小步验证、限制单次改动、先契约再编码**等做法，既能降低返工与线上风险，也有利于控制 AI 使用成本（减少无效多轮、减少大上下文请求）。因此「成本意识」不应以牺牲验收与契约为代价，而应与《AI-开发协作规范》中的协作原则一致。

---

## 4. 团队可采取的做法

- 在规范中明确：使用 AI 时需**兼顾效率与成本**，控制单次上下文与对话轮次，优先用需求卡与 rules 减少重复说明。
- 若使用按量计费 API，可约定用量上限或审批流程。
- 定期回顾：哪些场景用量高、是否可通过任务拆分与提示优化降低。
