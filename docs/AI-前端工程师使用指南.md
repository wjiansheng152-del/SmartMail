# AI-前端工程师使用指南（详细版）

本文档面向前端开发工程师，说明如何在真实项目中高效、安全地使用 AI，从需求澄清到联调回归形成闭环。

> 作者说明：本文档基于我当前阶段的前端协作经验整理。由于本人仍在成长阶段，某些建议可能偏向当前项目背景，对不同团队和业务场景未必完全适配。
>
> 欢迎你提出批评与指正，我会根据反馈不断修订，让文档更实用、更全面。

---

## 1. 使用目标

- 提升页面与交互开发效率。
- 降低前后端联调沟通成本。
- 减少因契约不清导致的返工。
- 保证体验质量与发布稳定性。
- 在使用 AI 时**兼顾效率与成本**：控制单次上下文与任务范围、明确需求以减少无效多轮，与「小步验证、限制单次改动」一致，也有利于控制用量。详见《AI开发的成本问题》。

---

## 2. 前端工程师与 AI 的分工

### 2.1 AI 负责

- 页面样板、列表与表单 CRUD 结构实现。
- API 封装、TypeScript 类型草稿、路由样板。
- 重复性代码迁移与规范化。
- 常见报错初步定位与修复建议。

### 2.2 工程师负责

- 交互流程与状态流设计。
- 用户体验与性能质量把关。
- 联调证据闭环与异常路径确认。
- 发布风险评估与上线验收。

---

## 3. 开发前必做事项

### 3.1 契约确认与需求对齐

- 动工前先与 AI 或团队对齐需求，减少理解偏差；可使用 **Cursor Plan 模式** 先产出计划再开发。
- 接口字段、分页结构、错误码语义。
- `id` 类型与来源（`local_id` / 内部主键）。
- 时间字段语义（UTC 输入输出规则）。
- 鉴权与租户头传递规则。

### 3.2 页面验收定义

每个页面至少定义：

- 正常流程 1 条（可操作并成功）。
- 异常流程 1 条（提示明确、可恢复）。
- 边界流程 1 条（空数据、超长、无权限等）。

---

## 4. 前端开发标准流程（可直接执行）

## 阶段 A：页面任务拆分

按页面拆成以下子任务：

- A1 数据获取与类型定义
- A2 视图布局与交互结构
- A3 表单提交与校验
- A4 异常处理与消息提示
- A5 回归与文档同步

## 阶段 B：AI 小步实现

**限制单次改动范围**，一次让 AI 实现一个子任务，不做跨页面大改；**逐步、逐接口验证**，通过后再进行下一项：

- 先完成 API 与类型，再做页面。
- 先完成列表，再做弹窗与编辑。
- 先做功能，再做细节优化。

## 阶段 C：单点验证

每个子任务后立刻验证：

- 页面是否可渲染、交互是否可用。
- 网络请求是否符合契约。
- 错误提示是否可感知且可理解。

## 阶段 D：联调闭环

用真实接口做验证，不只看 mock：

- 检查请求头、入参、响应结构。
- 关键链路做“接口响应 + 页面状态 + 数据结果”三方对照。
- 对关键字段进行端到端核验（如 `createdBy`、`runAt`）。

## 阶段 E：回归与收尾

- 清理临时调试代码与注释。
- 做核心路径回归（登录、列表、详情、创建、删除、查询）。
- 同步更新文档与变更说明。

## 5. 高质量提示词模板（给 AI）

### 5.1 页面开发模板

```markdown
目标：实现 xxx 页面（Vue3 + TS + Element Plus）。
接口：使用 /api/xxx，返回结构为 Result<IPage<T>>。
约束：
1) id 使用 local_id；
2) runAt 字段提交必须是 UTC；
3) 401 统一跳转登录；
4) 错误优先展示 errorInfo。
请输出：
- 需要新增/修改的文件列表
- 页面逻辑与关键状态说明
- 验证步骤（正常/异常/边界）
```

### 5.2 联调排障模板

```markdown
现象：
- 页面提示 xxx，接口状态码 xxx。
证据：
- 请求参数：
- 响应体：
- 控制台日志：
请给出：
1) 最可能根因列表（按优先级）
2) 每个根因的最小验证动作
3) 修复后需要回归的页面点
```

## 6. 前端场景下 AI 常见误区与缓解方式

**仍须重点注意的契约与体验类：**

- 把本地时间直接传后端，未做 UTC 转换。
- 误把内部主键当对外 id 使用。
- 列表页只处理成功态，忽略异常与空态。
- 捕获错误但不展示有效 `errorInfo`。

**可通过使用者操作缓解的：**

- **一次性大改多个页面**：限制单次 AI 改动范围，逐步、逐接口验证，避免一次改多页再集中回归。
- **需求理解偏差**：动工前先沟通清楚、冻结契约，可使用 Cursor **Plan 模式** 等对齐再开发。
- **知识滞后**：及时更新项目文档与依赖，或使用较新版本模型/工具。

**说明**：逻辑错误与复制粘贴残留、环境假设错误、敏感信息硬编码、API/库幻觉等，在当前主流模型下已明显缓解，可作为常规抽查而非默认高发风险。

## 7. 对前端 AI 使用者的具体要求（针对错误类型的防范）

针对 AI 可能产生的理解偏差、信息遗漏、任务执行错误、上下文丢失等，前端 AI 使用者须做到：

- **用 AI 之前**：动工前冻结可验收需求（含正常/异常/边界各至少 1 条）；在提示词或需求卡中显式约定 id 语义、时间（UTC/本地）、鉴权与错误码；复杂页面由人先拆成 A1～A5 等子任务；指令中写明改哪些文件、输出格式与必须交付物（如验证步骤）。
- **使用过程中**：一次只让 AI 完成一个子任务，不做跨页面大改；每完成一项就做单点验证（渲染、请求、错误提示），通过后再进行下一项；对接口路径、字段名、分页结构等与契约不一致处做核对；是否采纳 AI 的组件/状态设计方案由人决定。
- **拿到输出之后**：按「接口响应 + 页面状态 + 关键字段」做三方对照；验收至少覆盖正常、异常、边界流程；问题须有证据（请求/响应/控制台）再改；临时 console 或 mock 须登记并在收尾时清理。
- **长期**：接口与行为变更同步更新文档；关键约定（如 UTC、local_id）写入项目 rules 或需求卡；明确前端体验与联调结果的最终责任在使用者本人。

## 8. 前端质量门禁（DoD）

以下条件都满足，前端任务才算完成：

- 接口契约与字段语义已确认。
- 正常/异常/边界流程均有验证结果。
- 关键链路联调通过，证据可追溯。
- 错误提示可读且行为一致。
- 时间与 ID 语义无歧义。
- 临时调试逻辑已清理。
- 文档与变更说明已更新。

---

## 9. SmartMail 高相关实践建议

- 定时与立即发送相关页面必须统一 UTC 转换策略。
- 涉及活动与计划链路时，确认 `createdBy` 与用户身份一致。
- 统一使用项目 request 封装处理 token 刷新与异常展示。
- 页面报错时优先展示后端 `errorInfo`，减少“黑盒提示”。

## 10. 前端工程师进阶路径

- 从“让 AI 生成页面”升级到“让 AI 生成可验收页面”。
- 从“会改功能”升级到“会做联调证据闭环”。
- 从“功能完成”升级到“体验、性能、可维护性三者平衡”。
