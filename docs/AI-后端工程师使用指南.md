# AI-后端工程师使用指南（详细版）

本文档面向后端开发工程师，给出在真实项目中使用 AI 的可执行流程、提示词模板、质量门禁与风险清单。

> 作者说明：本文档是我基于当前项目经验做的阶段性总结。受限于本人目前的技术深度和项目阅历，部分方法可能更适用于当前规模与架构，不一定覆盖所有复杂场景。
>
> 欢迎批评指正，也欢迎补充你在生产环境中的最佳实践，我会持续迭代这份指南。

---

## 1. 使用目标

- 提升需求实现速度，减少样板代码投入。
- 降低联调返工率，缩短定位问题时间。
- 在不降低质量前提下提高交付稳定性。
- 在使用 AI 时**兼顾效率与成本**：控制单次上下文与任务范围、用需求卡与契约减少无效多轮，与「小步验证、限制单次改动」一致，也有利于控制用量。详见《AI开发的成本问题》。

---

## 2. 后端工程师与 AI 的分工

### 2.1 AI 负责

- 样板代码：Controller、DTO、VO、Mapper、Service 基础结构。
- 通用能力：参数校验、异常映射、统一返回结构。
- 文档与沉淀：接口文档草稿、变更说明、排障步骤整理。
- 初步排障：根据日志与错误栈给出排查路径。

### 2.2 工程师负责

- 领域建模、事务边界、幂等策略。
- 数据一致性与并发策略。
- 安全合规策略与生产风险决策。
- 回归验收与上线放行结论。

---

## 3. 开发前准备（必须）

### 3.1 需求冻结清单与需求对齐

动工前先充分沟通、对齐需求，减少理解偏差；可使用 **Cursor Plan 模式** 等先产出计划再开发。在让 AI 开始写代码前，至少冻结以下契约：

- 对外 `id` 语义（`local_id` 还是内部主键）。
- 时间字段语义（UTC 传输还是本地时间）。
- 鉴权头/租户头/用户头字段。
- 错误码与异常语义（400/401/403/404/409/422/500）。

### 3.2 环境事实清单

- 本次在 Docker 还是本地运行。
- 数据库地址、库名、账号、迁移脚本执行状态。
- 依赖服务是否可用（MQ、Redis、网关、IAM）。
- 当前分支与未提交改动情况。

## 4. 后端开发标准流程（可直接执行）

## 阶段 A：需求拆分

把需求拆分成可单独验收的任务：

- A1 接口契约变更任务
- A2 领域逻辑任务
- A3 数据结构/迁移任务
- A4 联调与回归任务
- A5 文档同步任务

每个任务都要有：输入、动作、输出、验证。

## 阶段 B：AI 实现单任务

**限制单次 AI 改动范围**，一次只让 AI 做一件事；**逐步、逐接口验证**，通过后再进行下一项，例如：

- 先只改后端接口，不同时改前端。
- 先只改查询链路，不同时改写入链路。

要求 AI 输出：

- 改动文件清单
- 关键逻辑说明
- 验证步骤

## 阶段 C：单点验证

每次改动后必须执行：

- 接口调用验证（至少 1 正常 + 1 异常）。
- SQL 对照验证（关键字段是否正确落库）。
- 日志检查（关键链路是否按预期执行）。

## 阶段 D：联调与回归

按主流程执行全链路：

- 鉴权 -> 业务写入 -> 异步处理 -> 查询结果

回归历史高风险点：

- `local_id` 解析
- `createdBy` 透传
- UTC 时间语义
- Docker profile 生效
- 迁移脚本遗漏

## 阶段 E：收尾与沉淀

- 清理临时调试代码。
- 更新接口文档与 BUGFIX 文档。
- 输出风险与回滚点说明。

## 5. 高质量提问模板（给 AI）

### 5.1 新增接口模板

```markdown
目标：新增 xxx 接口。
技术栈：Spring Boot + MyBatis-Plus。
约束：
1) 对外 id 使用 local_id；
2) 返回统一 Result；
3) 冲突返回 409；
4) 需要租户隔离。
请输出：
- 需要修改的文件清单
- 代码改动
- curl 验证命令
- SQL 验证语句
```

### 5.2 排障模板

```markdown
现象：
- 接口 xxx 返回 500，errorInfo=...
日志：
- 粘贴关键 20~50 行日志
环境：
- Docker / 本地
- DB 连接信息
请给出：
1) 最可能的 3 个根因（按概率排序）
2) 每个根因对应的最小验证动作
3) 修复建议和回归验证点
```

## 6. 后端场景下 AI 最常见误区与缓解方式

**仍须重点注意的契约与验证类：**

- 未明确 `id` 语义，导致跨服务查错数据。
- 忽略 UTC 语义，导致调度时间偏差。
- 只看接口状态码，不做 SQL 与日志交叉验证。
- 临时调试埋点未清理，污染主干代码。

**可通过使用者操作缓解的：**

- **只改代码不改迁移脚本**：在任务拆分时显式包含「迁移脚本与文档更新」，单步验证。当前 AI 对数据与迁移的配套已较以往更好，仍建议作为必检项。
- **需求理解偏差**：动工前先沟通清楚、冻结契约，可使用 Cursor **Plan 模式** 等对齐再开发。
- **一次性大改**：限制单次改动范围，逐步、逐接口验证。
- **知识滞后**：及时更新项目文档与依赖，或使用较新版本模型/工具。

**说明**：逻辑错误与复制粘贴残留、环境假设错误、敏感信息硬编码、API/库幻觉、权限越权等，在当前主流模型下已明显缓解，可作为常规抽查而非默认高发风险。

## 7. 对后端 AI 使用者的具体要求（针对错误类型的防范）

针对 AI 可能产生的理解偏差、逻辑推理错误、信息遗漏、任务执行错误、幻觉等，后端 AI 使用者须做到：

- **用 AI 之前**：动工前冻结可验收需求（含正常/异常/边界）；在需求卡或提示词中显式约定 id 语义、时间（UTC）、鉴权头与错误码；复杂需求由人先拆成接口/领域/迁移/联调/文档等任务，每项带验证动作；指令中写明改哪些文件、必须交付物（如 curl、SQL、迁移脚本）。
- **使用过程中**：一次只让 AI 做一件事（如只改一个接口或只改查询链路）；每完成一项就做单点验证（接口 + SQL + 日志），通过后再给下一项；对依赖版本、API 是否存在、配置项名称等做核对；事务边界、幂等与发布决策由人拍板。
- **拿到输出之后**：按接口、SQL、日志三重验证；验收至少 1 正常 + 1 异常，并覆盖关键边界；问题须有证据（响应/日志/SQL）再修；临时埋点须记录并在问题关闭后清理。
- **长期**：接口与行为变更同步更新文档；关键约定写入需求卡或 rules；主动更新给 AI 的项目知识（技术栈、禁止用法、部署方式）；明确后端正确性与线上责任的最终承担者为使用者本人。

---

## 8. 后端质量门禁（DoD）

满足以下条件才可认为后端任务完成：

- 契约冻结项全部有文档记录。
- 代码通过编译与最小回归。
- 接口、SQL、日志三重验证通过。
- 关键异常码符合约定。
- 临时调试代码已删除。
- 文档已同步更新（接口、状态、Bugfix）。

---

## 9. 场景化建议（SmartMail 高相关）

- 涉及活动、计划、投递链路时，必须同时验证 `createdBy` 传递。
- 涉及调度字段 `runAt` 时，必须明确 UTC 与展示转换边界。
- 涉及 local_id 改造时，必须验证查询与写入都按统一语义执行。
- Docker 场景下优先检查 profile、容器网络与数据库连接地址。

---

## 10. 后端工程师能力进阶建议

- 从“让 AI 写代码”升级到“让 AI 按契约写代码”。
- 从“报错了让 AI 看看”升级到“给证据让 AI 快速定位”。
- 从“完成功能”升级到“可回归、可追溯、可发布”。
