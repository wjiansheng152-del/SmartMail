# Cursor 使用说明书

本文档面向**使用 Cursor 进行开发的工程师**，提供可直接执行的操作说明、流程与模板。文中在适用处标注了可使用的 Cursor 功能：**[Plan]** 计划模式、**[Rules]** 规则、**[Skills]** 能力封装、**[Debug]** 调试/排障。

---

## 1. 适用范围与目标

### 1.1 适用对象

- 使用 **Cursor** 进行日常开发的前端、后端、测试、运维等岗位。
- 单人 + AI、多人 + AI、以及先开发后修复、并行子任务等协作形态。

### 1.2 使用目标

- **提效**：用 Cursor 加速样板代码、CRUD、文档与排障，同时控制单次上下文与任务范围，兼顾效率与成本。
- **可控**：通过契约冻结、小步验证、证据化排障，减少返工与线上风险。
- **可追溯**：需求可验收、改动可验证、问题有证据，文档与实现同频更新。

### 1.3 责任边界

- **AI 不承担最终业务正确性责任**，最终责任由 **AI 使用者（你）** 承担。
- 凡「出问题需要背责」的决策（发布、回滚、数据、安全）必须由岗位负责人主导，Cursor/AI 只可辅助。

## 2. Cursor 核心能力与推荐用法

### 2.1 Chat（对话）与 @ 引用

- **用法**：在 Chat 中用 `@文件名`、`@文件夹`、`@Codebase` 等把上下文限定在与当前任务相关的范围。
- **建议**：不要无差别 @ 整个仓库；只提供与当前任务相关的文件或目录，必要时用精简的接口/契约摘要代替大段代码，以控制 Token 与理解质量。
- **与规范的衔接**：满足「控制单次上下文」「只提供完成任务所必需的信息」的协作原则与安全最小暴露原则。

### 2.2 Plan 模式（计划模式）

- **何时用**：需求较复杂、涉及多模块或多步骤时，**先让 Cursor 产出计划再开发**，减少需求理解偏差。
- **建议**：动工前用 Plan 模式做一次「需求澄清 + 任务拆分」的预演，冻结可验收需求与契约后再进入实现。
- **标注**：**[Plan]** 适用于需求澄清与任务拆分阶段。

### 2.3 Rules（规则）

- **是什么**：项目或用户级规则，让 Cursor 在生成代码与建议时自动遵循既定规范。
- **建议写入的内容**：代码风格、项目目录结构、命名规则、API 调用规范、ID 语义（如 `local_id`）、时间语义（如 UTC）、鉴权头、错误码约定、禁止用法等。
- **好处**：减少每次对话中重复长说明，既提高一致性又有利于控制 Token；新对话或换人时上下文可复现。
- **标注**：**[Rules]** 适用于代码风格、契约、禁止用法等需全局生效的约定。

### 2.4 Skills（能力封装）

- **是什么**：把常见开发任务封装为标准能力，通过关键参数即可驱动，无需反复描述完整需求。
- **好处**：提升需求表达效率、减少 Token 消耗、简化与 Cursor 的协作方式。
- **标注**：**[Skills]** 适用于重复出现的开发任务（如生成表格页、接口封装模板等）。

### 2.5 多窗口/多会话

- **用法**：不同功能模块（如用户模块、订单模块）分窗口或分会话推进，避免单会话上下文过长与上下文污染。
- **与规范的衔接**：支持「限制单次改动范围、一次一事」的并行开发方式。

### 2.6 Debug（调试）**[Debug]**

Cursor 的 **Debug Mode** 是一种基于**运行时证据**的 Agent 调试模式：不先猜修复方案，而是先提出假设、加日志、收集运行数据，再根据真实运行信息定位根因并做**精准小改**（通常只需改两三行）。适合「能复现但原因不明显」的复杂 bug。

#### 2.6.1 何时使用 Debug Mode

- **回归问题**：以前正常，现在出问题，需要追踪变化。
- **性能问题与内存泄漏**：需要运行时 profiling 才能理解。
- **竞态与时序问题**：与执行顺序、异步行为相关。
- **能复现但难以从代码看出原因**：现象明确，根因不清晰。

当普通 Agent 对话难以解决时，Debug Mode 用「运行时证据」替代「猜测式修代码」，往往能可靠定位并修复。

#### 2.6.2 工作流程（官方流程）

1. **描述 bug**：在 Agent 中选择 Debug Mode，尽可能详细地描述问题（预期行为 vs 实际行为、复现步骤、报错信息、堆栈等）。
2. **探索与假设**：Agent 阅读相关代码，生成多个可能的根因假设。
3. **添加插桩（instrumentation）**：Agent 在关键路径插入日志语句，将运行数据发送到 Cursor 扩展内的本地 debug 服务（基于 HTTP，与语言无关）。
4. **复现 bug**：你按 Agent 给出的步骤在应用里复现问题，Agent 收集运行时的变量状态、执行路径、时序等信息。
5. **分析日志**：Agent 根据收集到的日志识别真实根因。
6. **精准修复**：Agent 做针对性修改，通常只改少量代码。
7. **验证并清理**：你再次复现以验证修复；确认无误后，Agent 移除所有插桩代码，只保留最小改动。

**人机闭环**：若修复不对或你不满意，可要求 Agent 增加日志、你再次复现，循环直到问题真正解决。

#### 2.6.3 Debug 的交付/输入

你只需提供：**对 bug 的描述**（现象、复现步骤、报错/堆栈等）或**复现后产生的日志/观测现象**。无需填写复杂的问题闭环或交付模板；问题闭环与交付模板见第 12 节，用于团队过程记录，与 Debug 功能无直接对应。

#### 2.6.4 使用建议（基于 Cursor 官方文档）

- **区分预期与实际**：明确写出「应该怎样」与「实际怎样」，便于 Agent 理解。
- **按步骤复现**：严格按 Agent 给出的复现步骤执行，保证日志能捕获到真实问题。
- **必要时多次复现**：对竞态、时序类问题，多次复现有助于 Agent 分析。
- **上下文尽量完整**：描述中包含错误信息、堆栈、具体操作步骤，有利于 Agent 设计更好的插桩。

#### 2.6.5 切换方式

- 在 Agent 的模式下拉菜单中选择 Debug Mode；或使用 **Shift+Tab** 快速切换模式。

## 3. 开发前必做事项（在 Cursor 中落地）

### 3.1 需求澄清与契约冻结

在让 Cursor 写代码之前，必须明确并**冻结**以下内容（可写在需求卡或第一条提示词里）：

- **ID 语义**：对外使用 `local_id` 还是内部主键，是否在查询与写入链路统一。
- **时间语义**：UTC 传输还是本地时间，前端展示与提交的转换边界（如 `runAt` 必须 UTC）。
- **鉴权与租户**：Token、租户头、用户头（如 `X-User-Id`）的传递规则；`createdBy` 在跨服务链路中的透传。
- **错误码与异常**：400/401/403/404/409/422/500 的语义及前端展示规则（如优先展示 `errorInfo`）。

**在 Cursor 中的做法**：**[Plan]** 使用 Plan 模式或第一条 Chat 消息中显式写出上述契约，并要求 Cursor 在实现时遵守；**[Rules]** 关键约定可写入 Rules，以便后续对话自动继承。

### 3.2 需求卡（标准输入模板）**[Plan]**

每次开发前，建议按下面模板整理需求，缺项标注「待定」，并将需求卡作为与 Cursor 对话的输入之一（可配合 **Plan** 做需求澄清）：

```markdown
## 需求名称

### 目标与范围
- 本期要做：
- 本期不做：

### 角色与场景
- 谁在什么场景下使用：

### 接口/数据契约
- 入参关键字段：
- 出参关键字段：
- ID 语义（internal id / local_id）：
- 时间语义（UTC / 本地时间）：
- 鉴权与租户信息：

### 验收标准
- 正常路径：
- 异常路径：
- 边界条件：

### 依赖与约束
- 依赖服务/数据库/消息队列：
- 环境约束（Docker、本地、CI）：

### 文档更新要求
- 需要更新哪些文档：
```

### 3.3 任务拆分与单任务指令

- **原则**：复杂需求由**人**先拆成单次可验证的小任务（每项有：输入、动作、输出、验证），再在 Cursor 中**一次只下达一个任务**。
- **在 Cursor 中的做法**：每个任务单独一条或一段提示，明确「改哪些文件」「输出格式」「必须交付物」（如 curl 命令、迁移脚本、验证步骤）；完成一项并**单点验证通过**后，再给下一项。

### 3.4 基线确认（阶段 0）

- 确认当前分支、未提交改动、运行环境（Docker/本地）。
- 确认数据库与依赖服务状态、是否存在未清理的临时调试代码。
- 在 Cursor 中若涉及环境或配置，可在提示词中写明「当前在 Docker 运行」或「本地运行」，避免 Cursor 按错误环境假设生成配置。

## 4. 标准开发流程（与 Cursor 操作对应）

以下流程与《AI-开发协作规范》一致，并标明在 Cursor 中如何执行。

| 阶段 | 动作 | 在 Cursor 中的操作 | 可用的 Cursor 功能 |
|------|------|---------------------|---------------------|
| **0. 启动前基线** | 确认改动边界、环境、未完成调试 | 在需求卡或首条消息中写明范围与环境 | — |
| **1. 需求澄清与契约冻结** | 冻结 ID/时间/鉴权/错误码；可验收需求 | 使用 Plan 或首条消息；关键约定写入 Rules | **[Plan]** **[Rules]** |
| **2. 任务拆分** | 拆成 T1/T2/…，每项带验证动作 | 在 Chat 中列出任务列表，一次只发一个任务给 Cursor | **[Plan]** |
| **3. AI 实现（小步循环）** | 单任务 → 单点验证 → 通过再下一项 | 每次只让 Cursor 做一件事；@ 只引用相关文件；验证通过后再发下一任务 | **[Rules]** **[Skills]**（若已封装） |
| **4. 联调与排障** | 接口 → 日志 → 数据库 证据链 | 把请求/响应/日志/SQL 粘贴或 @ 给 Cursor，要求根因与修复建议 | **[Debug]** |
| **5. 回归验证** | 核心主链路 + 历史高风险点 | 按清单自测，必要时把回归清单交给 Cursor 做检查建议 | **[Debug]**（辅助） |
| **6. 文档与沉淀** | 更新接口/启动/BUG 文档 | 在提示词中明确「需要更新 xxx 文档」并验收 | — |
| **7. 发布前检查** | 配置安全、依赖、迁移、镜像 | 发布决策由人做；可让 Cursor 帮助整理检查清单 | — |

**门禁**：未明确契约不编码；未拆出验证动作的任务不执行；未单点验证通过不进入下一任务；核心链路未通过不收尾；行为变更未更新文档不宣告完成。

## 5. 高质量提示词模板（直接用于 Cursor）

以下模板可直接在 Chat 中使用；若项目已用 **Rules** 约定契约，可简化约束部分；若任务可复用可考虑封装为 **Skills**。

### 5.1 前端页面开发 **[Rules]** **[Skills]**

```markdown
目标：实现 xxx 页面（Vue3 + TS + Element Plus）。
接口：使用 /api/xxx，返回结构为 Result<IPage<T>>。
约束：
1) id 使用 local_id；
2) runAt 字段提交必须是 UTC；
3) 401 统一跳转登录；
4) 错误优先展示 errorInfo。
请输出：
- 需要新增/修改的文件列表
- 页面逻辑与关键状态说明
- 验证步骤（正常/异常/边界）
```

### 5.2 后端新增接口 **[Rules]** **[Skills]**

```markdown
目标：新增 xxx 接口。
技术栈：Spring Boot + MyBatis-Plus。
约束：
1) 对外 id 使用 local_id；
2) 返回统一 Result；
3) 冲突返回 409；
4) 需要租户隔离。
请输出：
- 需要修改的文件清单
- 代码改动
- curl 验证命令
- SQL 验证语句
```

## 6. 验收与质量门禁（DoD）

### 6.1 前端完成定义

- 接口契约与字段语义已确认。
- 正常/异常/边界流程均有验证结果。
- 关键链路联调通过，证据可追溯。
- 错误提示可读且行为一致。
- 时间与 ID 语义无歧义。
- 临时调试逻辑已清理。
- 文档与变更说明已更新。

### 6.2 后端完成定义

- 契约冻结项全部有文档记录。
- 代码通过编译与最小回归。
- 接口、SQL、日志三重验证通过。
- 关键异常码符合约定。
- 临时调试代码已删除。
- 文档已同步更新（接口、状态、Bugfix）。

### 6.3 通用完成定义（协作规范）

- 需求清单与验收清单已冻结并有结果。
- 代码改动已通过单点验证与核心链路回归。
- 临时调试代码已清理。
- 相关文档已同步更新。
- 交付摘要已输出，风险已披露。

## 7. AI 可能产生的错误与在 Cursor 中的防范

### 7.1 错误类型与影响（概要）

| 错误类型 | 对项目的主要影响 |
|----------|------------------|
| 事实错误 | 技术选型被误导、排障被带偏、合规依据错误 |
| 幻觉 | 不存在的 API/配置/路径导致编译或部署失败 |
| 理解偏差 | 需求做偏、约束被忽略、验收与预期不一致 |
| 逻辑推理错误 | 业务规则写反、多步骤漏步或顺序错、并发/时序问题 |
| 上下文丢失 | 跨文件/跨模块不一致、技术栈或约定前后矛盾 |
| 知识过时 | 推荐废弃库或旧 API，技术债与安全风险 |
| 信息遗漏 | 实现缺步骤、异常/边界未覆盖、部署步骤不全 |
| 任务执行错误 | 忽略格式/范围/交付物要求，返工或风格混乱 |
| 复杂问题分解不足 | 大需求只实现一部分、缺乏系统结构、非功能需求被忽略 |

### 7.2 在 Cursor 中的防范要点

- **用 Cursor 之前**：**[Plan]** 在需求卡或首条消息中冻结可验收需求（正常/异常/边界）；显式约定 ID、时间、鉴权、错误码；复杂需求先拆成小任务；指令写清改哪些文件、输出格式、必须交付物；**[Rules]** 关键约定写入 Rules。
- **使用过程中**：一次只给一个任务；@ 只引用相关文件；每完成一项就做单点验证，通过后再发下一项；对依赖版本、API 是否存在、配置项名称等做核对；技术选型与发布决策由人拍板。
- **拿到输出之后**：**[Debug]** 按接口 + 数据 + 日志（或前端：接口 + 页面状态 + 关键字段）做验证；验收覆盖正常、异常、边界；问题须有证据再修；临时调试须登记并在问题关闭后清理。
- **长期**：文档与实现同频更新；**[Rules]** 关键约定写入需求卡或 Rules；主动更新项目文档供 Cursor 参考；明确最终责任在使用者本人。

## 8. 安全与合规（Cursor 使用场景）

- **配置与代码分离**：敏感信息（API 密钥、数据库密码、私钥）不写入代码，使用环境变量或配置中心；不在 Cursor 对话中粘贴真实生产密钥或完整生产数据。
- **最小暴露原则**：只向 Cursor 提供完成任务所**必需**的代码或配置片段；可脱敏或截取与当前任务相关的部分，不提供完整生产库、未脱敏日志、用户隐私数据。
- **依赖与权限**：引入 Cursor 建议的依赖前检查 CVE 与合规/版权；鉴权、权限、审计相关设计与实现由人做最终确认。
- **发布前检查**：确认无硬编码密钥、无调试用敏感输出、依赖无高危漏洞；若有组织安全流程，按流程评审。

## 9. 成本意识（使用 Cursor 时）

- **控制单次上下文**：不无差别 @ 整个仓库；只 @ 与当前任务相关的文件或目录，必要时用接口/契约摘要代替大段代码。
- **任务拆小、提示写清**：单任务、单次改动，减少「理解错了再重来」的轮次，降低 Token 与时间成本。
- **用好 Rules/Skills**：**[Rules]** **[Skills]** 把规范、契约、常用约束写进 Rules 或 Skills，避免每次对话大段重复说明。
- **按任务选模型**：简单补全、格式整理等可用更快/更省资源的模型；复杂设计、跨模块改动再用能力更强的模型（若 Cursor 支持多模型切换）。

小步验证、限制单次改动、先契约再编码，既利于质量也利于控制用量。

## 10. 岗位差异化速查

| 岗位 | Cursor 中优先让 AI 做 | 必须自己做的 | 可用的 Cursor 功能 |

## 11. 项目高频防错清单（Cursor 使用时必检）**[Rules]**

建议将以下清单中与项目强相关的条目写入 **Rules**，便于 Cursor 自动遵守；其余作为人工自检。

- **契约类**：`id` 是否统一约定（如 `local_id`）；`createdBy` 在计划与跨服务链路是否透传；用户头（如 `X-User-Id`）是否透传；时间字段是否统一约定（如 UTC），前端是否做本地时间换算。
- **环境类**：Docker 下数据库地址是否用服务名（如 `mysql`），而非 `localhost`；对应 profile 是否生效；迁移脚本是否已执行且与代码版本匹配。
- **验证类**：是否覆盖正常、异常、边界；关键接口是否用有效 Token；SQL 校验是否在正确库与租户下。

## 13. 小结：使用 Cursor 的十条铁律

1. **先契约再编码**：**[Plan]** **[Rules]** ID、时间、鉴权、错误码在首条消息或 Plan/Rules 中冻结。
2. **一次一事**：每次只让 Cursor 完成一个子任务或一个接口/页面，不跨模块大改。
3. **每步必验证**：每完成一项就做单点验证，通过后再给下一项。
4. **问题证据化**：**[Debug]** 排障时至少提供接口响应/日志/SQL 之一，再让 Cursor 分析修复。
5. **@ 引用要克制**：只 @ 与当前任务相关的文件或目录，控制上下文与成本。
6. **用好 Plan 与 Rules**：**[Plan]** **[Rules]** 复杂需求先用 Plan 对齐；规范与契约写入 Rules 减少重复。
7. **验收覆盖三类路径**：正常、异常、边界各至少一条。
8. **文档同频更新**：行为或接口变更必须同步更新文档。
9. **临时调试可回收**：**[Debug]** 排查用 log/断点/临时配置须登记，问题关闭后清理。
10. **责任在人**：最终业务正确性与发布/安全决策由 AI 使用者承担，不把 Cursor 结论当最终决策。

以上十条可直接作为使用 Cursor 时的自检清单。
