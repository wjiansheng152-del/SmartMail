# AI 可能产生的错误

本文档汇总 AI 在开发协作中可能产生的错误类型、对项目开发的影响，以及对使用者的防范要求概要。便于团队做风险识别与验收对照。

---

## 1. 错误类型与对项目开发的影响

| 错误类型 | 对项目开发的主要影响 |
|----------|------------------------|
| **事实错误** | 技术选型被误导、排障被带偏、合规依据错误。 |
| **幻觉** | 不存在的 API/配置/路径导致编译或部署失败。 |
| **理解偏差** | 需求做偏、约束被忽略、验收标准与预期不一致。 |
| **逻辑推理错误** | 业务规则写反、多步骤漏步或顺序错、并发/时序问题。 |
| **上下文丢失** | 跨文件/跨模块不一致、技术栈或约定前后矛盾。 |
| **知识过时** | 推荐废弃库或旧 API，留下技术债与安全风险。 |
| **偏见与数据偏差** | 默认假设不符合业务（如单时区、权限过松）。 |
| **过度自信** | 错误方案被当权威采纳、风险未被提示、责任边界模糊。 |
| **信息遗漏** | 实现缺步骤、异常/边界未覆盖、部署步骤不全。 |
| **错误泛化** | 从单场景泛化到全场景，忽略项目约束或容量差异。 |
| **任务执行错误** | 忽略格式/范围/交付物要求，导致返工或风格混乱。 |
| **复杂问题分解不足** | 大需求只实现一部分、缺乏系统结构、非功能需求被忽略。 |

---

## 2. 按影响类型归类

| 影响类型 | 主要相关的 AI 错误类型 |
|----------|------------------------|
| 功能错误/返工 | 幻觉、理解偏差、逻辑推理错误、信息遗漏、任务执行错误 |
| 架构/技术债/安全 | 事实错误、知识过时、错误泛化、偏见、过度自信 |
| 协作与一致性 | 上下文丢失、任务执行错误、复杂问题分解不足 |
| 交付与验收 | 信息遗漏、任务执行错误、复杂问题分解不足 |
| 合规与责任 | 事实错误、偏见、过度自信 |

---

## 3. 对 AI 使用者的防范要求概要

**用 AI 之前**：动工前冻结可验收需求（含正常/异常/边界）；显式约定 ID、时间、鉴权、错误码，新对话或新任务时重申；复杂需求由人先拆成单次可验证的小任务；指令具体、可检查（改哪些文件、输出格式、必须交付物）。

**使用过程中**：限制单次改动范围，一次一事；逐步、逐接口验证，通过后再给下一项；对版本号、依赖与 API 是否存在等关键事实做核对；技术选型与发布决策由人拍板，不把 AI 结论当最终决策。

**拿到输出之后**：按接口 + 数据 + 日志三重验证；验收至少覆盖正常、异常、边界各一类；问题必须证据化（响应/日志/SQL）再修复；临时调试须登记并在问题关闭后清理。

**长期与规范**：文档与实现同频更新；关键约定写入需求卡或 rules，保证上下文可复现；主动更新给 AI 的项目知识；明确「谁对结果负责」，发布与安全相关决策须责任人确认。

---

## 4. 与「已缓解」问题的关系

逻辑错误与复制粘贴残留、环境假设错误、数据与迁移遗漏、敏感信息硬编码、API/库幻觉、权限与越权等，在当前主流模型下已明显缓解，可作为常规抽查而非默认高发风险。需求理解偏差、一次性大改、知识滞后等，可通过流程与使用方式缓解，见《AI开发的局限》。
